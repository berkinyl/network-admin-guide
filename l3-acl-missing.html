Unauthorized Network Access Due to Missing or Misconfigured ACLs

Layer 3 – Network Layer

Overview

This tutorial explains a common Layer 3 security issue: failure to implement or correctly configure Access Control Lists (ACLs), resulting in unauthorized access between different network segments or VLANs. ACLs are critical for filtering traffic at Layer 3 based on source/destination IP addresses, protocols, and ports. Without ACLs, all routed traffic is allowed by default, which may violate security policies and expose sensitive systems.

Problem Scenario

In a typical routed network with inter-VLAN communication, devices in different subnets are allowed to communicate through a Layer 3 switch or router. By default, this routing is unrestricted. For instance, a workstation in VLAN 10 (192.168.10.0/24) can access a database server in VLAN 20 (192.168.20.0/24) unless explicitly denied.

Without ACLs to enforce segmentation, any device in one VLAN can initiate connections to any device in another VLAN. This exposes internal systems to unauthorized access, increases the attack surface, and can lead to data breaches or network compromise.

Common Symptoms and Risks

  • Users can reach services or servers outside their intended scope.
  • Sensitive applications are accessible from less secure zones.
  • Guest devices can ping or browse internal resources.
  • Security audits flag lack of logical isolation.
  • Network administrators lose visibility and control over inter-subnet flows.

Technical Background

Access Control Lists (ACLs) are filtering mechanisms that work at Layer 3 and 4. They allow or deny traffic based on rules matching IP addresses, port numbers, and protocols. Standard ACLs filter only by source IP, while extended ACLs provide more precise control using source/destination IP, protocol (TCP/UDP), and port.

By default, routers and Layer 3 switches forward traffic between all connected networks. ACLs must be explicitly configured to limit or block this behavior.

Step-by-Step Solution and Prevention

  • Define clear access policies: Decide which VLANs should communicate and which must be isolated.
  • Use extended ACLs for precision: Allow only specific traffic and deny the rest.
  • Apply ACLs correctly:
    • Inbound – on the interface receiving the traffic
    • Outbound – on the interface sending traffic to a destination
  • Example Configuration:
access-list 101 permit tcp 192.168.10.0 0.0.0.255 host 192.168.20.100 eq 80
access-list 101 deny ip any any
interface GigabitEthernet0/1.10
 ip access-group 101 in
  • Log denied attempts: Use log keyword for audit visibility.
  • Test the ACLs: Ping, traceroute, and scan to validate access.
  • Audit regularly: Clean up unused rules and document all ACLs.

Post-Fix Verification Steps

  • Verify only permitted traffic flows between VLANs.
  • Attempt unauthorized access and ensure it fails.
  • Use show access-lists to see hit counts.
  • Check logs for denied packets.

Best Practices

  • Follow the "default deny" principle.
  • Use consistent naming for numbered/named ACLs.
  • Group permissions where possible (object groups).
  • Apply ACLs close to the traffic source for better efficiency.
  • Document and comment all ACL rules clearly.

Conclusion

Access Control Lists are essential for enforcing logical security boundaries at Layer 3. Without them, all networks become interconnected by default, creating serious security vulnerabilities. Proper planning, accurate configuration, and continuous monitoring of ACLs help preserve isolation, reduce risks, and maintain a secure routing environment.
